### Title
Pool Mismatch Vulnerability in Momentum Position Valuation Allows Vault Accounting Manipulation

### Summary
The `update_momentum_position_value` function accepts an arbitrary pool parameter without validating it matches the position's `pool_id`, allowing anyone to calculate position values using a mismatched pool's price. This causes incorrect token amount calculations and corrupts the vault's USD valuation records, enabling attackers to manipulate share ratios for unauthorized fund extraction.

### Finding Description

The vulnerability exists in the `update_momentum_position_value` function where a position's value is calculated without verifying pool ownership. [1](#0-0) 

The function retrieves a `MomentumPosition` from the vault and calls `get_position_value` with the provided pool parameter. However, Momentum positions contain a `pool_id` field that identifies which pool they belong to: [2](#0-1) 

The position provides a getter for this pool_id: [3](#0-2) 

Similarly, pools expose their ID: [4](#0-3) 

**Root Cause:** In `get_position_token_amounts`, the function uses the provided pool's `sqrt_price` with the position's tick ranges and liquidity to calculate token amounts: [5](#0-4) 

No validation exists that `pool.pool_id() == position.pool_id()`. An attacker can provide Pool B while the position actually exists in Pool A, causing the liquidity math to calculate amounts based on Pool B's price with Pool A's position parameters.

**Why Existing Protections Fail:**

1. **No Authorization Required:** The function is public with no capability checks.

2. **Slippage Check Insufficient:** The slippage validation only ensures the pool price is within tolerance of the oracle price, not that it's the correct pool: [6](#0-5) 

3. **No Type-Level Enforcement:** `MomentumPosition` is not parameterized by pool ID, only by coin types, so multiple pools for the same token pair cannot be distinguished at the type level.

4. **Value Recording Is Blind:** The `finish_update_asset_value` function records whatever value is provided without validating its correctness: [7](#0-6) 

### Impact Explanation

**Direct Fund Impact:**
The corrupted position value directly affects the vault's total USD value calculation: [8](#0-7) 

This total value determines the share ratio used for all deposits and withdrawals: [9](#0-8) 

**Concrete Harm:**
1. **Inflated Valuation Attack:** Attacker provides a pool with higher price (within slippage) → position value inflated → total vault value inflated → share ratio inflated → attacker withdraws more funds than entitled
2. **Deflated Valuation Attack:** Attacker deflates position value → existing shareholders suffer losses as their share value decreases
3. **Manipulation Window:** Value updates persist until next correct update, allowing exploitation during deposit/withdrawal operations

**Affected Parties:**
- Existing vault shareholders lose funds through diluted or manipulated share values
- New depositors receive incorrect share amounts
- Vault protocol loses integrity of its core accounting mechanism

**Severity Justification:** Critical - allows unauthorized value extraction through accounting manipulation without requiring any privileged access.

### Likelihood Explanation

**Attacker Capabilities Required:**
- Access to create or reference a Momentum pool with same token types (permissionless in AMM protocols)
- Ability to call public functions (any Sui user)
- Knowledge of vault's position asset_type string (observable on-chain)

**Attack Complexity:** Low
1. Query vault to find Momentum position details
2. Create or identify alternative pool with same CoinA/CoinB types
3. Call `update_momentum_position_value` with wrong pool reference
4. Exploit manipulated share ratio through deposit or withdrawal

**Feasibility Conditions:**
- Vault must hold at least one Momentum position (normal operation)
- Attacker can create pools with arbitrary initial prices within slippage tolerance
- No special permissions or capabilities required

**Detection Constraints:**
- Value manipulation may appear legitimate as pool price passes slippage checks
- No on-chain events differentiate correct vs incorrect pool usage
- Impact only visible through comparison of calculated vs actual position values

**Probability:** High - attack is straightforward, low-cost, and directly exploitable by any external user.

### Recommendation

Add explicit pool ID validation in the `update_momentum_position_value` function:

```move
public fun update_momentum_position_value<PrincipalCoinType, CoinA, CoinB>(
    vault: &mut Vault<PrincipalCoinType>,
    config: &OracleConfig,
    clock: &Clock,
    asset_type: String,
    pool: &mut MomentumPool<CoinA, CoinB>,
) {
    let position = vault.get_defi_asset<PrincipalCoinType, MomentumPosition>(asset_type);
    
    // Add validation
    assert!(
        position.pool_id() == pool.pool_id(),
        ERR_POOL_MISMATCH
    );
    
    let usd_value = get_position_value(pool, position, config, clock);
    vault.finish_update_asset_value(asset_type, usd_value, clock.timestamp_ms());
}
```

**Additional Mitigations:**
1. Define new error constant: `const ERR_POOL_MISMATCH: u64 = 7_002;`
2. Add similar validation to other adaptor value update functions if they accept external pool/protocol references
3. Add integration test cases verifying rejection of mismatched pool references
4. Consider restricting value update functions to operator-only calls with proper capability checks

**Test Cases:**
1. Attempt to update position value with pool from different pool_id (should abort)
2. Verify correct pool ID passes validation
3. Test with multiple pools for same token pair
4. Verify value calculations remain correct with proper pool matching

### Proof of Concept

**Initial State:**
- Vault holds MomentumPosition in Pool_A (ID: 0xAAA) for SUI/USDC with tick range [100, 200] and liquidity L
- Pool_A current price: 0.995 SUI/USDC (sqrt_price: P_A)
- Oracle price: 1.0 SUI/USDC
- Slippage tolerance: 1%

**Attack Steps:**

1. **Attacker creates Pool_B** for SUI/USDC pair with initial price 1.005 SUI/USDC (sqrt_price: P_B, within 1% slippage tolerance)
   - Pool_B ID: 0xBBB

2. **Attacker calls:**
   ```
   update_momentum_position_value<PrincipalCoin, SUI, USDC>(
       vault_ref,
       oracle_config,
       clock,
       "MomentumPosition",  // position from Pool_A
       pool_B_ref           // wrong pool!
   )
   ```

3. **Function execution:**
   - Retrieves position from Pool_A (tick range [100, 200], liquidity L)
   - Uses Pool_B's sqrt_price (P_B) in `get_position_token_amounts`
   - Calculates amounts based on P_B with Pool_A's tick parameters
   - Pool price check passes (P_B within 1% of oracle)
   - Calculates incorrect USD value
   - Records incorrect value in vault.assets_value

**Expected vs Actual Result:**
- **Expected:** Position valued using Pool_A's actual price P_A
- **Actual:** Position valued using Pool_B's manipulated price P_B
- **Impact:** Total vault value incorrectly calculated, share_ratio manipulated

**Success Condition:** 
`vault.assets_value["MomentumPosition"]` reflects value calculated with Pool_B's price instead of Pool_A's actual price, enabling subsequent exploitation through deposit/withdrawal at manipulated share ratios.

```text
public fun get_position_token_amounts<CoinA, CoinB>(
    pool: &MomentumPool<CoinA, CoinB>,
    position: &MomentumPosition,
): (u64, u64, u128) {
    let sqrt_price = pool.sqrt_price();

    let lower_tick = position.tick_lower_index();
    let upper_tick = position.tick_upper_index();

    let lower_tick_sqrt_price = tick_math::get_sqrt_price_at_tick(lower_tick);
    let upper_tick_sqrt_price = tick_math::get_sqrt_price_at_tick(upper_tick);

    let liquidity = position.liquidity();

    let (amount_a, amount_b) = liquidity_math::get_amounts_for_liquidity(
        sqrt_price,
        lower_tick_sqrt_price,
        upper_tick_sqrt_price,
        liquidity,
        false,
    );
    (amount_a, amount_b, sqrt_price)
}
```

```text
public(package) fun finish_update_asset_value<PrincipalCoinType>(
    self: &mut Vault<PrincipalCoinType>,
    asset_type: String,
    usd_value: u256,
    now: u64,
) {
    self.check_version();
    self.assert_enabled();

    let last_update_time = &mut self.assets_value_updated[asset_type];
    *last_update_time = now;

    let position_value = &mut self.assets_value[asset_type];
    *position_value = usd_value;

    if (
        self.status() == VAULT_DURING_OPERATION_STATUS 
        && self.op_value_update_record.value_update_enabled 
        && self.op_value_update_record.asset_types_borrowed.contains(&asset_type)
    ) {
        self.op_value_update_record.asset_types_updated.add(asset_type, true);
    };
```

```text
public(package) fun get_total_usd_value<PrincipalCoinType>(
    self: &Vault<PrincipalCoinType>,
    clock: &Clock,
): u256 {
    self.check_version();
    self.assert_enabled();

    let now = clock.timestamp_ms();
    let mut total_usd_value = 0;

    self.asset_types.do_ref!(|asset_type| {
        let last_update_time = *self.assets_value_updated.borrow(*asset_type);
        assert!(now - last_update_time <= MAX_UPDATE_INTERVAL, ERR_USD_VALUE_NOT_UPDATED);

        let usd_value = *self.assets_value.borrow(*asset_type);
        total_usd_value = total_usd_value + usd_value;
    });

    emit(TotalUSDValueUpdated {
        vault_id: self.vault_id(),
        total_usd_value: total_usd_value,
        timestamp: now,
    });

    total_usd_value
}
```

```text
public(package) fun get_share_ratio<PrincipalCoinType>(
    self: &Vault<PrincipalCoinType>,
    clock: &Clock,
): u256 {
    self.check_version();
    self.assert_enabled();

    if (self.total_shares == 0) {
        return vault_utils::to_decimals(1)
    };

    let total_usd_value = self.get_total_usd_value(clock);
    let share_ratio = vault_utils::div_d(total_usd_value, self.total_shares);

    emit(ShareRatioUpdated {
        vault_id: self.vault_id(),
        share_ratio: share_ratio,
        timestamp: clock.timestamp_ms(),
    });

    share_ratio
}
```
