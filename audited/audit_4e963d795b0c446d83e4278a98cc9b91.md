# Audit Report

## Title
Pyth Oracle Failure Permanently Bricks Vault Operations with Suilend Positions

## Summary
When a vault has Suilend positions, any Pyth oracle failure (stale data >60s or confidence interval >10%) during vault operations permanently locks the vault in `VAULT_DURING_OPERATION_STATUS`, blocking all deposits and withdrawals with no admin recovery mechanism.

## Finding Description

The vulnerability stems from a critical mismatch between the Suilend oracle wrapper's graceful degradation design and its actual implementation in the price update flow.

**The Design Mismatch:**

The Suilend oracle wrapper `get_pyth_price_and_identifier()` is explicitly designed to return `Option::none()` when price validation fails, with comments stating "It returns None instead of aborting so the caller can handle invalid prices gracefully by eg falling back to a different oracle". [1](#0-0) 

The function returns None when confidence intervals exceed 10% of price [2](#0-1)  or when price data is more than 60 seconds stale. [3](#0-2) 

However, the actual caller `reserve::update_price()` immediately aborts with `EInvalidPrice` when receiving None, without implementing any fallback mechanism. [4](#0-3) 

**Complete DoS Execution Path:**

1. **Operation Initiation:** Operator calls `start_op_with_bag` which borrows `SuilendObligationOwnerCap` from the vault [5](#0-4)  and sets vault status to `VAULT_DURING_OPERATION_STATUS`. [6](#0-5) 

2. **Operation Completion Requirements:** After returning assets via `end_op_with_bag` [7](#0-6) , the operator must call `end_op_value_update_with_bag` which enforces that ALL borrowed assets have their values updated via `check_op_value_update_record`. [8](#0-7) 

3. **Zero Staleness Tolerance:** The value update check uses `PRICE_STALENESS_THRESHOLD_S = 0` (zero tolerance). [9](#0-8)  The Suilend adaptor enforces this by calling `assert_price_is_fresh()` on all deposit and borrow reserves. [10](#0-9) 

4. **Price Update Bottleneck:** To refresh prices, operators must call `lending_market::refresh_reserve_price` [11](#0-10)  which is the ONLY way to update the `price_last_update_timestamp_s` field that `assert_price_is_fresh` checks.

5. **Oracle Failure = Transaction Abort:** If Pyth returns None (confidence >10% or staleness >60s), the transaction aborts at `assert!(option::is_some(&price_decimal), EInvalidPrice)`, preventing any price update from completing.

6. **Permanent Lock:** Without completing value updates, `check_op_value_update_record` fails [12](#0-11) , preventing `end_op_value_update_with_bag` from executing and leaving the vault permanently stuck in `VAULT_DURING_OPERATION_STATUS`.

7. **Users Blocked:** Both deposits and withdrawals require `VAULT_NORMAL_STATUS` via `assert_normal()` checks. [13](#0-12) [14](#0-13) 

8. **No Admin Recovery:** The admin's `set_enabled` function explicitly blocks changes during operation status with `assert!(self.status() != VAULT_DURING_OPERATION_STATUS, ERR_VAULT_DURING_OPERATION)`. [15](#0-14)  The internal `set_status` function is only `public(package)` [16](#0-15)  and not exposed through any admin entry point in the manage module. [17](#0-16) 

## Impact Explanation

**Critical Protocol DoS with Permanent Fund Lock:**
- All vault deposits and withdrawals are permanently blocked, freezing user access to funds
- Complete loss of vault operational capability requiring contract upgrade (new package deployment) to recover
- All principal funds, DeFi positions (Suilend, Navi, Cetus, Momentum), pending requests, and accumulated fees become inaccessible
- Severe reputational damage and potential legal liability for locked user funds

This breaks the fundamental security guarantee that admin controls can recover from operational failures.

## Likelihood Explanation

**High Likelihood - Operational Failure Without Attack:**

This vulnerability triggers under realistic operational conditions:

1. **No Malicious Actor Required:** Normal vault operations by legitimate operators trigger the issue
2. **Realistic Trigger Conditions:** 
   - Pyth oracle confidence intervals >10% occur regularly during volatile markets
   - Price data >60 seconds stale happens during network congestion, oracle infrastructure issues, or Sui network delays
3. **Zero Mitigation:** No fallback oracle despite design intention, no staleness grace period (0 second threshold), no emergency admin recovery
4. **Architectural Weakness:** The oracle wrapper was designed for graceful degradation but the caller aborts instead of implementing the intended fallback logic

The combination of zero staleness tolerance and no fallback mechanism makes this highly likely to occur in production environments where oracle reliability issues are common.

## Recommendation

Implement a multi-layered fix:

1. **Immediate:** Add a fallback oracle mechanism in `reserve::update_price()` that uses the EMA price when spot price validation fails, matching the intended design from the oracle wrapper's comments.

2. **Short-term:** Introduce a configurable staleness grace period (e.g., 5-10 seconds) instead of zero tolerance to handle network delays.

3. **Long-term:** Add an admin emergency recovery function that can force-reset vault status from `VAULT_DURING_OPERATION_STATUS` to `VAULT_NORMAL_STATUS` with appropriate safeguards (e.g., requiring a time delay or multi-sig).

4. **Alternative:** Implement a vault operation timeout mechanism that automatically reverts to `VAULT_NORMAL_STATUS` after a configurable period (e.g., 1 hour) without completion.

## Proof of Concept

A complete test demonstrating this vulnerability would:

1. Create a vault with a Suilend position
2. Start an operation via `start_op_with_bag` (sets status to `VAULT_DURING_OPERATION_STATUS`)
3. Mock a Pyth oracle that returns None (simulating staleness >60s or confidence >10%)
4. Attempt to call `lending_market::refresh_reserve_price` → transaction aborts
5. Attempt to call `end_op_value_update_with_bag` → fails because value not updated
6. Attempt user deposit/withdraw → fails because vault not in `VAULT_NORMAL_STATUS`
7. Attempt admin `set_enabled` → fails with `ERR_VAULT_DURING_OPERATION`
8. Demonstrate vault is permanently stuck with no recovery path

The vulnerability is confirmed by the code paths shown in the citations above.

    vault: &mut Vault<T>,
    operation: &Operation,
    cap: &OperatorCap,
    mut defi_assets: Bag,
    tx: TxBag,
    principal_balance: Balance<T>,

                SuilendObligationOwnerCap<ObligationType>,
            >(

            );

                suilend_asset_type,
            );
            vault.return_defi_asset(suilend_asset_type, obligation);
        };

        if (defi_asset_type == type_name::get<MomentumPosition>()) {
            let momentum_asset_type = vault_utils::parse_key<MomentumPosition>(defi_asset_id);
            let momentum_position = defi_assets.remove<String, MomentumPosition>(
                momentum_asset_type,
            );
            vault.return_defi_asset(momentum_asset_type, momentum_position);
        };

            let receipt_asset_type = vault_utils::parse_key<Receipt>(defi_asset_id);
            let receipt = defi_assets.remove<String, Receipt>(receipt_asset_type);
            vault.return_defi_asset(receipt_asset_type, receipt);

        i = i + 1;

        vault.return_coin_type_asset<T, CoinType>(coin_type_asset_balance);
    } else {
        coin_type_asset_balance.destroy_zero();
    };

    vault.enable_op_value_update();

    defi_assets.destroy_empty();
}

public fun end_op_value_update_with_bag<T, ObligationType>(
    vault: &mut Vault<T>,
    operation: &Operation,
    cap: &OperatorCap,
    clock: &Clock,
    tx: TxBagForCheckValueUpdate,
) {
    vault::assert_operator_not_freezed(operation, cap);
    vault.assert_during_operation();

    let TxBagForCheckValueUpdate {
        vault_id,
        defi_asset_ids,
        defi_asset_types,
        total_usd_value,
        total_shares,
    } = tx;

    assert!(vault.vault_id() == vault_id, ERR_VAULT_ID_MISMATCH);

    // First check if all assets has been returned
    let length = defi_asset_ids.length();
    let mut i = 0;
    while (i < length) {
        let defi_asset_id = defi_asset_ids[i];
        let defi_asset_type = defi_asset_types[i];

        if (defi_asset_type == type_name::get<NaviAccountCap>()) {
            let navi_asset_type = vault_utils::parse_key<NaviAccountCap>(defi_asset_id);
            assert!(vault.contains_asset_type(navi_asset_type), ERR_ASSETS_NOT_RETURNED);
        };

        if (defi_asset_type == type_name::get<CetusPosition>()) {
            let cetus_asset_type = vault_utils::parse_key<CetusPosition>(defi_asset_id);
            assert!(vault.contains_asset_type(cetus_asset_type), ERR_ASSETS_NOT_RETURNED);
        };

        if (defi_asset_type == type_name::get<SuilendObligationOwnerCap<ObligationType>>()) {
            let suilend_asset_type = vault_utils::parse_key<
                SuilendObligationOwnerCap<ObligationType>,
            >(
                defi_asset_id,
            );
            assert!(vault.contains_asset_type(suilend_asset_type), ERR_ASSETS_NOT_RETURNED);
        };

        if (defi_asset_type == type_name::get<MomentumPosition>()) {
            let momentum_asset_type = vault_utils::parse_key<MomentumPosition>(defi_asset_id);
            assert!(vault.contains_asset_type(momentum_asset_type), ERR_ASSETS_NOT_RETURNED);
        };

        i = i + 1;
    };

    let total_usd_value_before = total_usd_value;
    vault.check_op_value_update_record();
    let total_usd_value_after = vault.get_total_usd_value(
        clock,
    );

    // Update tolerance if there is a loss (there is a max loss limit each epoch)
    let mut loss = 0;
    if (total_usd_value_after < total_usd_value_before) {
        loss = total_usd_value_before - total_usd_value_after;
        vault.update_tolerance(loss);
    };

    assert!(vault.total_shares() == total_shares, ERR_VERIFY_SHARE);

    emit(OperationValueUpdateChecked {
        vault_id: vault.vault_id(),
        total_usd_value_before,
        total_usd_value_after,
        loss,
    });

    vault.set_status(VAULT_NORMAL_STATUS);
    vault.clear_op_value_update_record();
}

    self.check_version();

        current_deposit_id,
        receipt_id,
        recipient,
        self.id.to_address(),
        amount,
        expected_shares,
        clock.timestamp_ms(),
    );
    self.request_buffer.deposit_requests.add(current_deposit_id, new_request);

    emit(DepositRequested {
        request_id: current_deposit_id,
        receipt_id: receipt_id,
        recipient: recipient,
        vault_id: self.id.to_address(),
        amount: amount,
        expected_shares: expected_shares,
    });

    // Operator will retrieve this coin and execute the deposit
