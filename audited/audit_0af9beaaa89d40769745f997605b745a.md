### Title
Volo Vault Withdrawal Denial of Service Due to Insufficient Free Principal Liquidity

### Summary
Similar to the Juiced Protocol vulnerability, the Volo Vault system allows users to request withdrawals but cannot execute them when `free_principal` balance is insufficient because vault assets are deployed to external DeFi protocols (Navi, Suilend, Cetus, Momentum). There is no automatic mechanism to retrieve liquidity from these external positions, causing withdrawal executions to fail until operators manually rebalance liquidity, creating a denial of service for legitimate withdrawal requests.

### Finding Description

The Volo Vault implements a two-phase withdrawal system where users first request withdrawals [1](#0-0) , and operators later execute them [2](#0-1) .

The vault stores user deposits in two forms:
1. **`free_principal`** - immediately available liquidity [3](#0-2) 
2. **`assets` Bag** - DeFi positions from external protocols like Navi, Suilend, Cetus, and Momentum [3](#0-2) 

During operations, operators can borrow free principal to deploy to external yield-generating protocols [4](#0-3) . The vault also manages DeFi position objects that represent funds deployed to external protocols [5](#0-4) .

**Root Cause:** When executing withdrawals, the vault checks if sufficient `free_principal` is available [6](#0-5) . If this check fails, the withdrawal execution aborts with `ERR_NO_FREE_PRINCIPAL`. However, at the request phase, there is NO liquidity check [7](#0-6) , allowing users to create withdrawal requests that cannot be immediately executed.

**Why Protections Fail:** There is no automatic mechanism to pull back assets from Navi, Suilend, Cetus, or Momentum positions when withdrawal demand exceeds available `free_principal`. Operators must manually:
1. Start an operation to borrow DeFi positions [5](#0-4) 
2. Withdraw/liquidate positions from external protocols using adaptors
3. Return freed liquidity to the vault [8](#0-7) 
4. Complete the operation

Only after this manual intervention can pending withdrawal requests be executed.

### Impact Explanation

**High Severity Denial of Service:**
- During periods of high withdrawal demand or when most vault assets are deployed to external protocols, users cannot complete withdrawals
- Withdrawal requests remain stuck in the `request_buffer` until operators manually rebalance
- Users have no guarantee when their withdrawal will become executable
- This breaks the core protocol invariant that users should be able to withdraw their funds
- While funds are not permanently lost, the temporary inaccessibility creates significant operational risk and user harm

The vault's test suite explicitly acknowledges this vulnerability with a dedicated test case [9](#0-8)  that expects `ERR_NO_FREE_PRINCIPAL` when free principal is insufficient.

### Likelihood Explanation

**High Likelihood - Normal Operating Conditions:**
1. **No Attacker Required:** This occurs during normal vault operations when assets are deployed for yield generation
2. **Common Scenario:** Vaults naturally deploy most user deposits to external protocols (Navi, Suilend, Cetus, Momentum) to generate yield, leaving minimal `free_principal`
3. **User-Triggered:** Any user can request a withdrawal at any time [1](#0-0) 
4. **No Preconditions:** Users only need valid receipt shares to request withdrawals
5. **Market Conditions:** During market volatility or yield-seeking periods, most vault liquidity is naturally deployed to external positions
6. **Operator Dependency:** The vault relies entirely on operator availability and responsiveness to manually rebalance liquidity

The vulnerability is realistic and can occur without malicious intent during normal vault operations.

### Recommendation

Implement one or both of the following mitigations:

**Option 1: On-Chain Automatic Rebalancing**
Modify `execute_withdraw` to automatically liquidate external positions when `free_principal` is insufficient:
```
If (amount_to_withdraw > self.free_principal.value()) {
    // Automatically initiate operation to withdraw from external positions
    // Start with most liquid positions (Cetus, then Navi/Suilend)
    // Pull back only the minimum needed to satisfy withdrawal
}
```

**Option 2: Liquidity Reserve Requirement**
Add a minimum `free_principal` threshold that must be maintained:
- Before deploying assets to external protocols, ensure sufficient `free_principal` remains
- Add check in `borrow_free_principal` [4](#0-3)  to prevent depleting reserves below threshold
- Set threshold based on historical withdrawal patterns (e.g., 10% of total vault value)

**Option 3: Liquidity Check at Request Time**
Add a check in `request_withdraw` [7](#0-6)  to validate that either sufficient `free_principal` exists OR emit a warning event so users know their withdrawal may be delayed.

### Proof of Concept

**Setup:**
1. Vault has 10,000 tokens total value
2. 9,000 tokens deployed to Navi/Suilend positions for yield
3. 1,000 tokens remain as `free_principal`

**Exploit Steps:**
1. User A requests withdrawal of 5,000 tokens worth of shares [1](#0-0) 
   - Request succeeds, added to `request_buffer`
2. Operator attempts to execute User A's withdrawal [10](#0-9) 
   - Execution fails at line 1036: `assert!(5000 <= 1000, ERR_NO_FREE_PRINCIPAL)` âŒ
3. User A's withdrawal is blocked until operator:
   - Starts operation [5](#0-4) 
   - Borrows Navi/Suilend positions from vault
   - Withdraws 4,000 tokens from external protocols
   - Returns liquidity to `free_principal` [11](#0-10) 
   - Completes operation [8](#0-7) 
4. Only after manual rebalancing can User A's withdrawal be executed

This exact scenario is tested in the codebase [12](#0-11) , demonstrating the vulnerability is a known limitation rather than being protected against.

public(package) fun borrow_free_principal<PrincipalCoinType>(
    self: &mut Vault<PrincipalCoinType>,
    amount: u64,
): Balance<PrincipalCoinType> {
    self.check_version();
    self.assert_enabled();

    if (self.status() == VAULT_DURING_OPERATION_STATUS) {
        let principal_asset_type = type_name::get<PrincipalCoinType>().into_string();
        self.op_value_update_record.asset_types_borrowed.push_back(principal_asset_type);
    };

    let ret = self.free_principal.split(amount);
    emit(FreePrincipalBorrowed {
        vault_id: self.vault_id(),
        amount: amount,
    });
    ret
}

public(package) fun request_withdraw<PrincipalCoinType>(
    self: &mut Vault<PrincipalCoinType>,
    clock: &Clock,
    receipt_id: address,
    shares: u256,
    expected_amount: u64,
    recipient: address,
): u64 {
    self.check_version();
    self.assert_normal();
    assert!(self.contains_vault_receipt_info(receipt_id), ERR_RECEIPT_NOT_FOUND);

    let vault_receipt = &mut self.receipts[receipt_id];
    assert!(vault_receipt.status() == NORMAL_STATUS, ERR_WRONG_RECEIPT_STATUS);
    assert!(vault_receipt.shares() >= shares, ERR_EXCEED_RECEIPT_SHARES);

    // Generate request id
    let current_request_id = self.request_buffer.withdraw_id_count;
    self.request_buffer.withdraw_id_count = current_request_id + 1;

    // Record this new request in Vault
    let new_request = withdraw_request::new(
        current_request_id,
        receipt_id,
        recipient,
        self.id.to_address(),
        shares,
        expected_amount,
        clock.timestamp_ms(),
    );
    self.request_buffer.withdraw_requests.add(current_request_id, new_request);

    emit(WithdrawRequested {
        request_id: current_request_id,
        receipt_id: receipt_id,
        recipient: recipient,
        vault_id: self.id.to_address(),
        shares: shares,
        expected_amount: expected_amount,
    });

    vault_receipt.update_after_request_withdraw(shares, recipient);

    current_request_id
}

public(package) fun execute_withdraw<PrincipalCoinType>(
    self: &mut Vault<PrincipalCoinType>,
    clock: &Clock,
    config: &OracleConfig,
    request_id: u64,
    max_amount_received: u64,
): (Balance<PrincipalCoinType>, address) {
    self.check_version();
    self.assert_normal();
    assert!(self.request_buffer.withdraw_requests.contains(request_id), ERR_REQUEST_NOT_FOUND);

    // Get the current share ratio
    let ratio = self.get_share_ratio(clock);

    // Get the corresponding withdraw request from the vault
    let withdraw_request = self.request_buffer.withdraw_requests[request_id];

    // Shares and amount to withdraw
    let shares_to_withdraw = withdraw_request.shares();
    let usd_value_to_withdraw = vault_utils::mul_d(shares_to_withdraw, ratio);
    let amount_to_withdraw =
        vault_utils::div_with_oracle_price(
            usd_value_to_withdraw,
            vault_oracle::get_normalized_asset_price(
                config,
                clock,
                type_name::get<PrincipalCoinType>().into_string(),
            ),
        ) as u64;

    // Check the slippage (less than 100bps)
    let expected_amount = withdraw_request.expected_amount();

    // Negative slippage is determined by the "expected_amount"
    // Positive slippage is determined by the "max_amount_received"
    assert!(amount_to_withdraw >= expected_amount, ERR_UNEXPECTED_SLIPPAGE);
    assert!(amount_to_withdraw <= max_amount_received, ERR_UNEXPECTED_SLIPPAGE);

    // Decrease the share in vault and receipt
    self.total_shares = self.total_shares - shares_to_withdraw;

    // Split balances from the vault
    assert!(amount_to_withdraw <= self.free_principal.value(), ERR_NO_FREE_PRINCIPAL);
    let mut withdraw_balance = self.free_principal.split(amount_to_withdraw);
