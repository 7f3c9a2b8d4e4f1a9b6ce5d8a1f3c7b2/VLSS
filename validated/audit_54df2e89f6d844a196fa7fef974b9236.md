# Audit Report

## Title
Momentum Position Valuation Excludes Uncollected Trading Fees Leading to Systematic Undervaluation

## Summary
The Momentum adaptor's position valuation logic systematically excludes uncollected trading fees (`owed_coin_x` and `owed_coin_y`) from position value calculations, causing all Momentum positions to be undervalued. This directly impacts the vault's share price calculation, leading to unfair share distribution during deposits (users receive excess shares) and withdrawals (users receive insufficient principal).

## Finding Description

The Momentum Position struct contains separate tracking for principal liquidity and uncollected trading fees [1](#0-0) , with getter functions available to retrieve these fee amounts [2](#0-1) .

However, the `get_position_token_amounts()` function only retrieves the position's liquidity amount and converts it to token amounts using mathematical formulas, completely ignoring the `owed_coin_x` and `owed_coin_y` fields [3](#0-2) .

This incomplete valuation flows through the system:

1. The undervalued token amounts are used directly in USD value calculation [4](#0-3) 

2. The undervalued USD amount is stored in the vault's assets_value table [5](#0-4)  via [6](#0-5) 

3. The vault's total USD value calculation sums these undervalued amounts [7](#0-6) 

4. The share ratio (share price) is calculated as total_usd_value / total_shares, resulting in an artificially low ratio [8](#0-7) 

This breaks the fundamental accounting invariant that vault share price must accurately reflect all vault assets, including accrued but uncollected value.

## Impact Explanation

**Direct Fund Misallocation:**

The undervalued share ratio directly affects user fund distribution:

- **Deposits**: Users receive shares calculated as `new_usd_value / share_ratio` [9](#0-8) . Since share_ratio is too low, users receive more shares than they should, diluting existing holders.

- **Withdrawals**: Users receive principal calculated based on `shares * ratio` [10](#0-9) . Since ratio is too low, users receive less principal than their shares are worth.

**Cumulative Effect**: As trading fees accumulate on active Momentum positions (which happens automatically with any swap activity in the pool), the valuation gap grows larger over time. High-volume positions could accumulate significant fees, leading to substantial misvaluation.

**Affected Parties**:
- New depositors gain unfair advantage (receive excess shares)
- Withdrawers suffer losses (receive insufficient principal)  
- Existing holders are diluted
- Protocol TVL reporting is systematically incorrect

## Likelihood Explanation

**Certainty: 100% - This is a systematic accounting bug, not an attack vector.**

**Automatic Trigger**: Trading fees accumulate automatically on any Momentum liquidity position whenever swaps occur in the pool. The Momentum protocol tracks these fees in the `owed_coin_x` and `owed_coin_y` fields, which can be collected via `mmt_v3::collect::fee()` [11](#0-10) . However, the Volo vault never collects or accounts for these fees in valuation.

**No Attack Required**: Every call to `update_momentum_position_value()` will produce an undervalued result whenever the position has any uncollected fees, which is the normal state for active positions.

**Preconditions**: Only requires the vault to hold a Momentum position with trading activity - standard operational conditions.

## Recommendation

Add the uncollected fees to the token amounts returned by `get_position_token_amounts()`:

```move
public fun get_position_token_amounts<CoinA, CoinB>(
    pool: &MomentumPool<CoinA, CoinB>,
    position: &MomentumPosition,
): (u64, u64, u128) {
    let sqrt_price = pool.sqrt_price();
    
    let lower_tick = position.tick_lower_index();
    let upper_tick = position.tick_upper_index();
    
    let lower_tick_sqrt_price = tick_math::get_sqrt_price_at_tick(lower_tick);
    let upper_tick_sqrt_price = tick_math::get_sqrt_price_at_tick(upper_tick);
    
    let liquidity = position.liquidity();
    
    let (amount_a, amount_b) = liquidity_math::get_amounts_for_liquidity(
        sqrt_price,
        lower_tick_sqrt_price,
        upper_tick_sqrt_price,
        liquidity,
        false,
    );
    
    // Add uncollected trading fees to get complete position value
    let owed_a = position.owed_coin_x();
    let owed_b = position.owed_coin_y();
    
    (amount_a + owed_a, amount_b + owed_b, sqrt_price)
}
```

This ensures that all value belonging to the vault's position, including both principal liquidity and accrued fees, is included in the valuation calculation.

## Proof of Concept

Due to the Momentum protocol functions being stub implementations in the local dependencies (all functions abort), a direct executable test cannot be provided. However, the vulnerability is evident from static code analysis:

1. Position struct has `owed_coin_x` and `owed_coin_y` fields that track uncollected fees
2. Getter functions exist to retrieve these values
3. `get_position_token_amounts()` never calls these getters
4. No other code path in the vault adds these fees to the valuation
5. The undervalued amounts flow directly into share ratio calculations

The logic chain is deterministic: incomplete valuation → undervalued total_usd_value → incorrect share_ratio → unfair share distribution.

```text
public fun get_position_value<CoinA, CoinB>(
    pool: &MomentumPool<CoinA, CoinB>,
    position: &MomentumPosition,
    config: &OracleConfig,
    clock: &Clock,
): u256 {
    let (amount_a, amount_b, sqrt_price) = get_position_token_amounts(pool, position);

    let type_name_a = into_string(get<CoinA>());
    let type_name_b = into_string(get<CoinB>());

    let decimals_a = config.coin_decimals(type_name_a);
    let decimals_b = config.coin_decimals(type_name_b);

    // Oracle price has 18 decimals
    let price_a = vault_oracle::get_asset_price(config, clock, type_name_a);
    let price_b = vault_oracle::get_asset_price(config, clock, type_name_b);
    let relative_price_from_oracle = price_a * DECIMAL / price_b;

    let pool_price = sqrt_price_x64_to_price(sqrt_price, decimals_a, decimals_b);
    let slippage = config.dex_slippage();
    assert!(
        (pool_price.diff(relative_price_from_oracle) * DECIMAL  / relative_price_from_oracle) < (DECIMAL  * slippage / SLIPPAGE_BASE),
        ERR_INVALID_POOL_PRICE,
    );

    let normalized_price_a = vault_oracle::get_normalized_asset_price(config, clock, type_name_a);
    let normalized_price_b = vault_oracle::get_normalized_asset_price(config, clock, type_name_b);

    let value_a = vault_utils::mul_with_oracle_price(amount_a as u256, normalized_price_a);
    let value_b = vault_utils::mul_with_oracle_price(amount_b as u256, normalized_price_b);

    value_a + value_b
}
```

```text
public(package) fun get_total_usd_value<PrincipalCoinType>(
    self: &Vault<PrincipalCoinType>,
    clock: &Clock,
): u256 {
    self.check_version();
    self.assert_enabled();

    let now = clock.timestamp_ms();
    let mut total_usd_value = 0;

    self.asset_types.do_ref!(|asset_type| {
        let last_update_time = *self.assets_value_updated.borrow(*asset_type);
        assert!(now - last_update_time <= MAX_UPDATE_INTERVAL, ERR_USD_VALUE_NOT_UPDATED);

        let usd_value = *self.assets_value.borrow(*asset_type);
        total_usd_value = total_usd_value + usd_value;
    });

    emit(TotalUSDValueUpdated {
        vault_id: self.vault_id(),
        total_usd_value: total_usd_value,
        timestamp: now,
    });

    total_usd_value
}
```
